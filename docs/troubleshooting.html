Troubleshooting

The issues students actually hit, with the fixes. Commands below use the C++ IDE's names — for the x86 IDE, swap in unlv-x86-ide and port 8218.

"Cannot connect to the Docker daemon" or "docker: command not found"

Docker Desktop isn't running (or isn't installed). Launch Docker Desktop and wait until the whale icon in your menu bar or system tray reports it's running, then retry the command. Docker Desktop must be running any time you start or use an IDE.

"Port is already allocated" or the port is in use

Something else on your machine is using the IDE's port. Change only the left side of the port mapping — for example -p 127.0.0.1:8136:8080 — and open that port in your browser instead (http://127.0.0.1:8136). The right side, 8080, is inside the container and must stay as is.

"The container name is already in use"

The container already exists from a previous run, so you don't need the long docker run command again — just start it:

docker start unlv-cpp-ide

If you actually want a fresh container (for example, mid-update), remove the old one first with docker rm unlv-cpp-ide, then rerun the full command. Your files are safe either way — they live in your workspace folder, not in the container.

Files seem to be missing after an update

Your files are not gone — they're in ~/UNLV/cpp-workspace (or ~/UNLV/x86-workspace) on your own computer; check that folder in Finder or Explorer. If the IDE opens to an empty or different workspace, the new container was started with a different -v path than before. Rerun the command with the exact same -v "$HOME/UNLV/cpp-workspace:/home/coder/workspace" mapping and everything reappears.

Never drop the -v part of the run command — it's what keeps your work in a normal folder on your machine instead of inside the container.

x86 IDE on an Apple Silicon Mac

The x86 image is x86-64 only, by design — Docker Desktop runs it under emulation on Apple Silicon. Assembling, linking, and running all work (a little slower). Include --platform linux/amd64 in the run command to silence the platform warning; forgetting it is harmless.

The one real limitation: gdb cannot debug under emulation, and a gdb script may look like it succeeded while its output file contains only labels with no values. If an assignment's deliverable is gdb output, produce it on a native x86-64 machine. Details on the x86 IDE page.

The IDE makes my computer slow

Two things help on modest hardware. First, stop the container when you're done working — closing the browser tab does not stop it: docker stop unlv-cpp-ide. Second, cap the IDE's resources by adding --memory=2g --cpus=2 right after docker run -it in the run command; a runaway program then can't take down the rest of your machine. Turning off Docker Desktop's "start at login" setting also helps.

Still stuck?

Ask in class or office hours, or open an issue on the GitHub repository with the exact command you ran and the full error message.
